M3-05.0B Architecture & Version Foundation Plan
Purpose
M3-05.0B will improve Adventure Companion's internal architecture without changing the current production experience. The work will be incremental, GitHub Pages compatible, and reversible through focused commits.
Production baseline
Starting point: `main` after merge of M3-05.0A
Working branch: `m3-05-0b-architecture-version-foundation`
Current released experience: M3-04.4C — Experience Completion
Runtime model: static HTML, CSS, and browser JavaScript hosted through GitHub Pages
Build tooling: none required for production

Findings
1. Build information needs one source of truth
`app.js` currently defines application build metadata near the beginning of the file. Reliability and diagnostics also need access to build identity. M3-05.0B should establish one browser-safe source of truth that can be consumed without a bundler.
2. Configuration and experience data are intermingled
`app.js` includes dates, phase metadata, addresses, navigation destinations, itinerary stops, reservations, readiness items, dashboard content, storage keys, and rendering logic. These should not be separated all at once. M3-05.0B will first extract only low-risk shared metadata and constants.
3. GitHub Pages compatibility limits the migration strategy
Adventure Companion must continue running as a static site. The architecture foundation must therefore avoid introducing Node-only runtime dependencies, build steps, server requirements, or import patterns that are not deliberately supported by the existing page-loading strategy.
4. Existing feature modules are useful boundaries
Packing, reliability, weather service, and weather UI already have dedicated files. M3-05.0B should strengthen these boundaries rather than perform a broad rewrite.

Approved build sequence
Build 0 — Architecture plan
Deliver this document only. No runtime files change.
Build 1 — Centralized version metadata
Create a single browser-safe version object and connect existing build/diagnostics displays to it.
Required properties should include:
milestone/build identifier;
human-readable name;
release status;
build or release date;
cache/version identifier where appropriate.
The implementation must preserve the current visible build information and diagnostics behavior.
Build 2 — Shared constants inventory and first extraction
Identify repeated values and extract only low-risk constants. Candidates include:
local-storage keys;
trip date boundaries;
phase identifiers and labels;
weather cache/refresh timing;
diagnostics initialization timing.
No itinerary, reservation, traveler, or Smart Stop content will be redesigned in this build.
Build 3 — Module contract documentation
Document the responsibilities and public interfaces of:
core app orchestration;
packing;
reliability;
weather service;
weather UI.
Any code movement must be separately approved after this contract is documented.
Migration rules
Preserve all current functionality.
Maintain GitHub Pages compatibility without requiring a production build step.
Make one architectural change per commit.
Keep existing runtime filenames in place unless a move is explicitly approved.
Do not combine architecture work with itinerary or visual redesign.
Do not modify traveler accomplishments or celebration order.
Bubbe and Papa remain full adventurers.
Cades Cove remains excluded from all data, plans, examples, and fallback content.
Individual accomplishments continue to be celebrated before family accomplishments.
Continue using change packages containing `README_FIRST.md` and `CHANGED_FILES.md` for implementation builds.
Verification requirements
Every implementation build must pass all three gates:
Code review — confirm scope, loading order, public interfaces, and no accidental data changes.
Automated testing — run all existing JavaScript tests plus focused tests for the new version/config behavior.
Manual visual verification — confirm the dashboard, daily adventures, weather, packing, diagnostics, and navigation render and operate normally.
GitHub Pages must not be described as verified until it is manually checked after deployment.
Risks and controls
Risk	Control
Script loading-order failure	Add one shared script at a time and verify it loads before consumers
Service-worker cache serving mixed versions	Change cache identifiers deliberately and test update behavior
Global-name collision in browser scripts	Use one clearly namespaced, immutable application configuration object
Hidden behavior change during extraction	Compare before/after values and retain focused regression tests
Broad refactor becoming difficult to review	Keep each pull request narrow and independently reversible
Trip content accidentally altered	Treat itinerary/traveler data as out of scope for version-foundation commits
Success criteria
M3-05.0B is complete when:
one authoritative version source is used by the application and diagnostics;
approved shared constants are centralized without changing their values;
module responsibilities and dependencies are documented;
all existing automated tests pass;
focused new tests pass;
manual visual verification passes;
the deployed GitHub Pages application is personally verified after merge;
no prohibited itinerary content or family-rule regressions are introduced.
Out of scope
Adventurer profiles and accomplishments data model;
new user-facing features;
visual redesign;
itinerary changes;
packing recommendation changes;
historical-document cleanup;
test-file relocation;
conversion to a framework or bundler;
service-worker redesign beyond version/cache coordination required by this foundation.
Decision checkpoint
Approval of this document authorizes Build 1 planning and implementation of centralized version metadata. It does not authorize broad movement or rewriting of `app.js`.
